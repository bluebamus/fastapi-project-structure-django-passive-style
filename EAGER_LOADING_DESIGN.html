Eager-Loading 독립성 유지 설계안

도메인 앱의 탈착 가능성(INSTALLED_APPS pluggability)을 지키면서, BaseRepository의 eager-loading 8메서드를 실사용·검증되는 살아있는 코드로 전환하는 설계.

대상: app/core/repositories/repository_base.py 원칙: intra-domain relationship 예시: blog · Post 1:N Comment 성격: 기능추가(비파괴)

🎯
한 줄 요약. 자연 관계가 전부 도메인 간이라 관계를 배선하면 앱 독립성이 깨진다. 해법은 한 도메인 안에서 닫히는 관계(예: blog의 Post 1:N Comment)를 하나 도입하는 것 — 그러면 8메서드가 실제로 쓰이면서도 FK가 도메인 경계를 넘지 않아 탈착성이 유지된다.

01문제 정의

BaseRepository는 N+1 쿼리를 막기 위한 eager-loading 메서드 8종을 제공한다. 그러나 이들은 SQLAlchemy relationship()이 정의돼 있어야 동작하는데, 현재 어떤 도메인 모델에도 관계가 없어 전부 호출 불가능한 "스캐폴딩" 상태다.

eager-loading 8메서드 (관계 필요)

메서드목적현재
get_by_id_withPK 단건 + 관계 로드미사용
get_one_with조건 단건 + 관계 로드미사용
get_by_ids_with다중 ID(IN) + 관계 로드미사용
get_many_with필터 목록 + 관계 로드미사용
get_all_with전체 목록 + 관계 로드미사용
get_in_batches대용량 배치 스트림부분
get_with_join명시적 JOIN + contains_eager미사용
count_with_relation관계 개수 집계미사용

02근본 원인 — 자연 관계가 전부 "도메인 간"

도메인 모델에는 관계가 될 만한 필드가 실제로 존재한다. 하지만 전부 느슨한 문자열 참조(FK/relationship 없음)이며, 가리키는 대상이 다른 도메인이다.

그림 2-1 · 현재의 암묵적 관계는 모두 도메인 경계를 가로지른다 (점선 = 느슨한 문자열 참조)

⚠️
딜레마. 이 관계들에 ForeignKey+relationship()을 걸면 eager-loading은 쓸 수 있게 되지만, 예컨대 reply.post_id → blog_posts.id FK가 생겨 reply 앱이 blog 앱에 DB 레벨로 결합된다. 그러면 "blog를 빼면 reply 스키마가 깨지는" 상태가 되어 INSTALLED_APPS 탈착성이라는 이 템플릿의 정체성과 정면충돌한다.

03설계 원칙 — Intra-Domain Relationship

✅
원칙. 관계를 한 도메인 내부에서 닫히도록 만든다. 관계의 양쪽 테이블이 같은 도메인 소유이면, FK가 그 도메인 자기 테이블끼리만 걸려 다른 도메인과 결합이 생기지 않는다.

04핵심 설계 · 데이터 모델

blog 도메인 내부에 1:N 관계를 하나 추가한다: 게시글(Post) 1 : N 댓글(Comment). 두 테이블 모두 blog 소유다.

그림 4-1 · ER 다이어그램 (blog 도메인 내부)
erDiagram
  POST ||--o{ COMMENT : "comments (1:N)"
  POST {
    string  id PK "UUIDMixin"
    string  title
    text    content
    string  author "느슨참조(문자열)"
    datetime created_at "TimestampMixin"
  }
  COMMENT {
    string  id PK "UUIDMixin"
    string  post_id FK "→ blog_posts.id (도메인 내부)"
    text    content
    datetime created_at "TimestampMixin"
  }

모델 스케치

# app/domains/blog/models/models.py  (동일 도메인 · 동일 패키지)
from sqlalchemy import ForeignKey, Text
from sqlalchemy.orm import Mapped, mapped_column, relationship
from app.core.db.session import Base
from app.core.models.models_base import TimestampMixin, UUIDMixin

class Post(Base, UUIDMixin, TimestampMixin):
    __tablename__ = "blog_posts"
    title:   Mapped[str]
    content: Mapped[str] = mapped_column(Text)
    author:  Mapped[str | None]                      # 기존 느슨참조 유지
    # ▼ intra-domain 관계
    comments: Mapped[list["Comment"]] = relationship(
        back_populates="post", cascade="all, delete-orphan",
    )

class Comment(Base, UUIDMixin, TimestampMixin):
    __tablename__ = "blog_comments"
    post_id: Mapped[str] = mapped_column(
        ForeignKey("blog_posts.id"), index=True,     # FK는 blog 내부에서만 성립
    )
    content: Mapped[str] = mapped_column(Text)
    post: Mapped["Post"] = relationship(back_populates="comments")

05동작 흐름 — eager-loading이 실제로 하는 일

계층은 기존 아키텍처(Router → Service → Repository → DB)를 그대로 따른다. 아래는 get_by_id_with(post_id, ["comments"])를 selectin 전략으로 호출했을 때의 실제 흐름이다.

그림 5-1 · 시퀀스 — 게시글 상세 + 댓글(N+1 없이 2쿼리)
sequenceDiagram
  autonumber
  actor Client
  participant RT as "Router (blog/v1)"
  participant SV as "BlogService"
  participant RP as "PostRepository"
  participant DB as "Database"
  Client->>RT: GET /api/v1/blog/posts/{id}?with_comments=true
  RT->>SV: get_post_with_comments(id)
  SV->>RP: get_by_id_with(id, ["comments"], "selectin")
  RP->>DB: SELECT * FROM blog_posts WHERE id = :id
  DB-->>RP: Post 1행
  RP->>DB: SELECT * FROM blog_comments WHERE post_id IN (:id)
  DB-->>RP: Comment N행
  RP-->>SV: Post (post.comments 로드 완료)
  SV-->>RT: PostDetail 스키마
  RT-->>Client: 200 OK (JSON)
  Note over RP,DB: 총 2쿼리 — 댓글 접근 시 추가 쿼리 없음
🧭
N+1이 왜 사라지나? 순진한 구현은 게시글 1건을 읽고, 화면에서 post.comments를 만질 때마다 매번 SELECT ... WHERE post_id = ?를 날린다(글이 많으면 N번). selectin은 관련 댓글을 한 번의 IN (...) 쿼리로 미리 가져와 이 반복을 없앤다.

078메서드 실사용 매핑 (blog Post/Comment)

하나의 intra-domain 관계만으로 8메서드 전부를 실제 사용·테스트할 수 있다.

#메서드blog에서의 구체 용도실행 SQL 개요
1get_by_id_with게시글 상세 + 댓글posts WHERE id + comments WHERE post_id IN(id)
2get_one_withslug 등 조건으로 글 1건 + 댓글posts WHERE slug + comments IN(...)
3get_by_ids_with여러 글 + 각 댓글(N+1 제거)posts WHERE id IN(...) + comments IN(...)
4get_many_with작성자 필터 목록 + 댓글posts WHERE author LIMIT + comments IN(...)
5get_all_with전체 글 목록 + 댓글posts LIMIT/OFFSET + comments IN(...)
6get_in_batches대량 글 배치 처리(집계/마이그레이션)posts LIMIT :n OFFSET :k 반복
7get_with_join댓글 달린 글만 조인 조회posts JOIN comments ON post_id=id
8count_with_relation글별 댓글 수 통계posts LEFT JOIN comments GROUP BY id
그림 7-1 · 시퀀스 — count_with_relation("comments") (집계 1쿼리)
sequenceDiagram
  autonumber
  participant SV as "BlogService"
  participant RP as "PostRepository"
  participant DB as "Database"
  SV->>RP: count_with_relation("comments")
  RP->>DB: SELECT posts.*, COUNT(comments.id) AS cnt
FROM blog_posts LEFT JOIN blog_comments
ON posts.id = comments.post_id GROUP BY posts.id
  DB-->>RP: [(Post, cnt), ...]
  RP-->>SV: [(글, 댓글수), ...]

검증 항목결과

교육적 이점. reply는 "cross-domain 느슨참조(독립성 우선)", blog.Comment는 "intra-domain 관계(ORM 기능 시연)"로 남아, 템플릿이 두 가지 결합 전략을 동시에 예시하게 된다.

09부록 — 로딩 전략 비교 (selectin / joined / subquery)

같은 "관계 로드"라도 strategy에 따라 실행 SQL이 다르다. 1~5번 메서드가 공통으로 받는 파라미터다.

전략쿼리 수방식권장 상황
selectin (기본)2본체 조회 후 … WHERE fk IN (…)1:N 컬렉션(댓글 등)
joined1LEFT JOIN 한 방에1:1 · N:1 (단일 연관)
subquery22번째를 서브쿼리로selectin 유사, 특정 상황
그림 9-1 · selectin(2쿼리) vs joined(1쿼리)
flowchart LR
  subgraph SI["selectin (2쿼리)"]
    A1["① posts 조회"] --> A2["② comments
WHERE post_id IN(...)"]
  end
  subgraph JO["joined (1쿼리)"]
    B1["① posts LEFT JOIN comments"]
  end

10대안 비교

방향eager-loading 사용도메인 독립성비용/리스크평가

C. 관계 미도입 + eager-loading 삭제 불가(삭제) 유지 표면 최소화, 추후 재작성 기능 상실

11구현 체크리스트 · 영향 범위

1
모델 — blog/models에 Comment 추가, Post.comments/Comment.post 관계 정의. (믹스인 UUIDMixin/TimestampMixin 재사용)
2
레포지토리 — 기존 PostRepository가 BaseRepository의 eager-loading을 그대로 상속·사용(추가 코드 최소).
3
서비스 — get_post_with_comments, list_posts_with_comment_counts 등에서 get_by_id_with/count_with_relation 호출.
4
스키마·라우터(선택) — 댓글 포함 상세, 글별 댓글수 등 신규 엔드포인트(기존 계약 불변, 추가만).
5
마이그레이션 — DEBUG는 create_all이 처리, 운영은 Alembic 리비전 1건.
6
테스트 — eager-loading 실경로 + 글별 댓글수 + (선택)중첩 로딩 검증. 현재 미검증 경로를 실측 커버.
📦
영향 요약. 성격은 기능추가(비파괴). 기존 엔드포인트·응답 스키마는 불변이고 추가만 발생한다. 도메인 독립성·INSTALLED_APPS 탈착성은 그대로 보존된다.
